## 2. كشف الاحتيال لكل طبقة

### 2.1 طبقة API (Application Layer)

```
🚨 Rate Limiting:
┌─────────────────────────────────────────────────────┐
│  المسار            │ الحد             │ الإجراء     │
├─────────────────────┼──────────────────┼─────────────┤
│ POST /auth/login    │ 5/دقيقة/IP       │ Block 15 د  │
│ POST /auth/register │ 3/ساعة/IP        │ Block 30 د  │
│ POST /orders        │ 10/دقيقة/مستخدم  │ Block 60 د  │
│ POST /orders/:id/photo│ 5/دقيقة/مستخدم │ Warn + Block│
│ POST /payments      │ 5/10 دقائق/مستخدم│ Flag        │
│ GET /api/*          │ 100/دقيقة/IP     │ Throttle    │
└─────────────────────┴──────────────────┴─────────────┘

⛔ Block vs Throttle vs Flag:
├── Block → يمنع الطلب كلياً (يرجع 429 Too Many Requests)
├── Throttle → يبطئ الطلب (يزيد وقت الاستجابة)
└── Flag → يسمح لكن يسجل تحذير في الـ Logs

🚦 Brute Force Protection:
├── بعد 5 محاولات فاشلة لتسجيل الدخول:
│   ├── قفل الحساب لمدة 15 دقيقة
│   ├── إشعار للمستخدم: "محاولة دخول مشبوهة"
│   └── تسجيل IP في Blacklist المؤقت
├── بعد 10 محاولات فاشلة:
│   ├── قفل الحساب يدوياً (يتطلب مراجعة)
│   └── إشعار للأدمن
└── الـ OTP نفسه:
    ├── 3 محاولات خاطئة → إعادة إرسال OTP جديد
    └── 5 محاولات خاطئة → قفل 30 دقيقة
```

### 2.2 طبقة الصور (Photo Layer)

```
🔍 فحص الصور:

نظام الـ Hash:
├── كل صورة مرفوعة → SHA-256 Hash
├── مقارنة مع قاعدة بيانات الصور السابقة
├── إذا نفس الـ Hash → طلب مكرر
└── إذا Hash مشابه (95%+) → صورة معدّلة قليلاً

كشف تزوير الإيصالات (Receipt Fraud):
├── AI يفحص صورة الإيصال (المرحلة الثانية):
│   ├── هل التوقيع معدّل؟
│   ├── هل المبلغ مزور؟
│   ├── هل الخلفية مختلفة (مقص من تطبيق آخر)؟
│   └── هل الإيصال من تطبيق أصلية؟
│
├── يدوي (المرحلة الأولى):
│   الموظف يتحقق:
│   1. هل الإيصال من تطبيق كريمي/جيب الأصلي؟
│   2. هل الأرقام متناسقة؟
│   3. هل اسم المستلم = Qor3a؟
│   4. هل المبلغ = 1,000 (غير معدّل)؟
│
└── 🚩 علامات الخطر:
    ├── إيصال بدقة أقل من 480p
    ├── إيصال بدون رقم عملية
    ├── إيصال بتاريخ مختلف عن اليوم
    └── إيصال من حساب غير معروف
```

### 2.3 طبقة الحسابات (User Layer)

```
مكافحة الحسابات الوهمية:

نظام تسجيل الدخول:
├── OTP → رقم الهاتف يجب أن يكون حقيقياً (يرسل رمز)
├── كل رقم هاتف → حساب واحد فقط
└── إذا حاول رقم ثاني ← "هذا الرقم مسجل مسبقاً"

نظام كشف تعدد الحسابات:
├── نفس الـ IP مع حسابات متعددة:
│   ├── 2-3 حسابات من IP واحد → طبيعي (عائلة)
│   ├── 5-10 حسابات → تحذير (مكتب؟)
│   └── 10+ حسابات → حظر (Bot)
│
├── نفس الجهاز (Device Fingerprint):
│   ├── نستخدم canvas fingerprint أو simple headers
│   └── إذا 3+ حسابات من نفس الجهاز → Flag
│
├── نفس الصورة الشخصية (SHA-256):
│   └── نفس الصورة في طلبين → الثاني مرفوض
│
└── أنماط مشبوهة:
    ├── كل الحسابات من نفس المدينة
    ├── كل الحسابات سجلت في نفس الساعة
    └── كل الحسابات بدون بيانات عائلة (unusual)
```

### 2.4 طبقة الموظف (Employee Layer)

```
مكافحة احتيال الموظف:

كل إجراء للموظف مسجّل:
├── 🆔 معرف الموظف
├── 📋 نوع الإجراء (تأكيد دفع، تغيير حالة، تعديل بيانات)
├── 🕒 التوقيت بالضبط
├── 🌐 الـ IP
└── 📸 Screenshot (اختياري — في الإجراءات الحرجة)

صلاحيات محدودة:
├── الموظف:
│   ├── ✅ يقرأ الطلبات
│   ├── ✅ يؤكد الدفع
│   ├── ✅ يغير الحالة (ضمن المسموح)
│   └── ❌ لا يعدّل بيانات العميل
│
├── الأدمن:
│   ├── ✅ يقرأ كل الطلبات
│   ├── ✅ يعدّل أي شيء
│   ├── ✅ يلغي الطلبات
│   └── ✅ يشوف التقارير
│
└── قيود إضافية:
    ├── الموظف لا يستطيع تأكيد دفعة لـ (نفسه/قريبه)
    ├── أي تغيير في الطلب يحتاج audit trail
    └── لا يمكن حذف السجلات (soft delete فقط)

🚩 علامات الخطر للموظف:
├── موظف يؤكد 50 دفعة في ساعة واحدة (مشبوه)
├── موظف يغير حالة طلبات من نفس المنطقة
├── موظف يعمل خارج أوقات الدوام المتوقعة
└── موظف يعدّل طلبات بعد تأكيدها
```

## 3. خوارزمية كشف الاحتيال (Rule Engine)

### 3.1 القواعد الأساسية

```
Rule 1: "نفس الصورة في طلبين"
  IF photo_hash IN previous_photos
  THEN reject("هذه الصورة مستخدمة في طلب آخر")

Rule 2: "نفس الـ IP مع أكثر من 5 حسابات"
  IF ip_account_count[req.ip] > 5
  THEN flag("حسابات متعددة من نفس الـ IP")

Rule 3: "إيداع سريع (أقل من 30 ثانية بين الخطوات)"
  IF step_time < 30s
  THEN flag("Bot-like behavior — إدخال سريع جداً")

Rule 4: "نفس رقم واتساب لطلبين"
  IF phone IN verified_orders
  THEN reject("هذا الرقم مسجل مسبقاً")

Rule 5: "مبلغ غير 1,000 في الإيصال"
  IF receipt_amount ≠ 1000
  THEN reject_with_message("المبلغ يجب أن يكون 1,000 ريال")

Rule 6: "إيصال بنفس رقم العملية"
  IF transaction_id IN old_transactions
  THEN reject("هذا الإيصال مستخدم سابقاً — احتيال محتمل")

Rule 7: "موظف يؤكد 30+ دفعة في الساعة"
  IF employee_daily_confirmations > 30_per_hour
  THEN flag("نشاط غير طبيعي للموظف — مراجعة")

Rule 8: "حساب جديد يسجل 10 طلبات في أول يوم"
  IF account_age < 1_day AND order_count > 10
  THEN flag("حساب مشبوه — طلبات كثيرة من حساب جديد")
```

## 4. نظام الـ Flag Levels

### 4.1 مستويات التحذير

| المستوى | اللون | المعنى | الإجراء |
|---------|-------|--------|---------|
| L0 | 🟢 | طبيعي | لا شيء |
| L1 | 🟡 | مشبوه قليلاً | مراجعة بشرية عشوائية |
| L2 | 🟠 | مشبوه | مراجعة بشرية إجبارية |
| L3 | 🔴 | احتيال مؤكد | رفض تلقائي + حظر |
| L4 | 🟣 | احتيال خطير | حظر + إبلاغ السلطات |

### 4.2 ماذا يحدث لكل مستوى

```
L0 — 🟢 طبيعي:
├── الطلب يكمل بشكل طبيعي
└── لا مراجعة إضافية

L1 — 🟡 مشبوه قليلاً:
├── يُضاف إلى قائمة "مراجعة عشوائية"
├── موظف يراجع 10% من هذه الطلبات
├── إذا أكّد الموظف → L0
└── إذا رفض الموظف → L2

L2 — 🟠 مشبوه:
├── يُضاف إلى قائمة "مراجعة إجبارية"
├── موظف يراجع 100% من هذه الطلبات
├── يتخذ قرار (قبول/رفض)
└── مسجل في audit log

L3 — 🔴 احتيال مؤكد:
├── رفض تلقائي
├── الحساب محظور (لا يستطيع تسجيل الدخول)
├── إشعار للأدمن
└── إشعار للعميل: "تم رفض طلبك بسبب مخالفة الشروط"

L4 — 🟣 احتيال خطير:
├── كل ما في L3 +
├── حظر الـ IP
├── حظر الجهاز
├── إبلاغ الجهات المختصة (إن لزم)
└── توثيق للقضية
```

## 5. الـ Audit Trail

### 5.1 ماذا نسجل

```
كل شيء:
├── 👤 من قام بالإجراء (user_id, role)
├── 🕐 متى (timestamp)
├── 🌐 من أين (IP, User-Agent)
├── 📋 ماذا فعل (action, resource, old_value, new_value)
└── 🔍 لماذا (reason, flag_level)

الجدول في قاعدة البيانات:
┌─────────────────────────────────────────────────┐
│ audit_logs                                       │
├─────────────────────────────────────────────────┤
│ id: UUID (PK)                                     │
│ user_id: UUID (FK → users)                        │
│ action: VARCHAR(50) [verify_payment, change_status│
│           , update_profile, cancel_order...]      │
│ resource_type: VARCHAR(50) [order, payment, user] │
│ resource_id: UUID                                 │
│ old_value: JSONB                                  │
│ new_value: JSONB                                  │
│ ip_address: INET                                  │
│ user_agent: TEXT                                  │
│ flag_level: INT [0-4]                             │
│ metadata: JSONB                                   │
│ created_at: TIMESTAMPTZ                           │
└─────────────────────────────────────────────────┘
```
