# 🧑‍💼 اختبار قبول المستخدم — User Acceptance Testing (UAT)

> **Qor3a (قرعة)** — بروتوكول UAT، سيناريوهات، معايير القبول

---

## 1. نظرة عامة

```
هدف UAT: التحقق من أن النظام يلبي احتياجات المستخدمين الحقيقية
الفئة: عميل + موظف + مدير
المدة: أسبوع واحد
عدد المشاركين: 10-20
النتيجة المرجوة: SUS Score > 70
```

## 3. سيناريوهات UAT

### 3.1 Client Scenarios (للمشاركين P1-P4)

#### سيناريو 1: التسجيل الكامل (الرحلة السعيدة)
```
الوقت: 5-10 دقائق
التعليمات:
1. افتح الموقع على جوالك
2. سجل برقم هاتفك
3. أدخل رمز OTP المرسل
4. املأ البيانات الشخصية (5 خطوات)
5. التقط صورتك (حاول تخليها صح)
6. اختر طريقة الدفع (كريمي)
7. ارفع صورة إيصال (حتى لو وهمي)
8. سجل الخروج

ماذا نراقب:
□ هل الخطوات واضحة؟
□ كم استغرقت؟
□ هل واجهت صعوبة في أي خطوة؟
□ هل فهمت كل حقل؟

نتيجة القبول:
□ أكمل 100% من الخطوات بدون مساعدة → ناجح
□ احتاج مساعدة في خطوة واحدة → مقبول
□ احتاج مساعدة في أكثر من خطوة → يحتاج تحسين
```

#### سيناريو 2: الصورة المرفوضة
```
التعليمات:
1. سجل الدخول
2. ابدأ طلب جديد
3. التقط صورة بخلفية غير بيضاء (مثلاً: في المطبخ)

ماذا نراقب:
□ هل فهمت سبب الرفض؟
□ هل عرفت كيف تصلحه؟
□ هل أعادت التصوير بسهولة؟

نتيجة القبول:
□ أصلح الصورة من أول مرة → ناجح
□ احتاج محاولتين → مقبول
□ 3+ محاولات → يحتاج تحسين
```

#### سيناريو 3: تتبع الطلب
```
التعليمات:
1. ادخل حسابك
2. اعرض طلباتك
3. اقرأ حالة آخر طلب
4. اعرض تفاصيل الطلب

ماذا نراقب:
□ هل وجدت الطلب بسهولة؟
□ هل فهمت الحالة؟
□ هل تعرف ماذا تفعل بعدين؟

نتيجة القبول:
□ وجد وفهم كل شيء → ناجح
□ احتاج توضيح للحالة → مقبول
□ ما فهم شيء → يحتاج تحسين
```

#### سيناريو 4: فحص النتيجة
```
التعليمات:
1. ادخل حسابك
2. اختر طلب مكتمل
3. اضغط "فحص النتيجة"
4. شاهد النتيجة

ماذا نراقب:
□ هل فهمت الزر؟
□ هل النتيجة واضحة؟
□ هل تعرف الخطوات القادمة؟

نتيجة القبول:
□ عرف يفحص وفهم النتيجة → ناجح
□ طلب توضيح للنتيجة → مقبول
□ ما عرف يفحص → يحتاج تحسين
```

### 3.2 Employee Scenarios (للمشارك P5)

#### سيناريو 5: مراجعة الطلبات
```
التعليمات:
1. ادخل لوحة الموظفين
2. اعرض طلبات "payment_verification"
3. اختر طلباً
4. راجع البيانات + الصورة + الإيصال
5. غيّر الحالة إلى "approved" مع ملاحظات

ماذا نراقب:
□ هل القائمة سهلة التصفية؟
□ هل كل المعلومات في شاشة واحدة؟
□ هل تغيير الحالة سريع؟

نتيجة القبول:
□ أنجز المراجعة في < دقيقتين → ناجح
□ 2-4 دقائق → مقبول
□ > 4 دقائق → يحتاج تحسين
```

#### سيناريو 6: إدخال رقم التأكيد
```
التعليمات:
1. اختر طلباً بحالة "approved"
2. ادخل رقم التأكيد الذي حصلت عليه من الموقع الرسمي
3. أرسل

نتيجة القبول:
□ الإدخال سريع وواضح → ناجح
□ بطيء أو مربك → يحتاج تحسين
```

### 3.3 Admin Scenarios (للمشارك P6)

#### سيناريو 7: لوحة الإدارة
```
التعليمات:
1. ادخل لوحة المدير
2. اعرض الإحصائيات
3. ابحث عن مستخدم
4. غيّر صلاحية مستخدم
5. صدّر تقرير CSV

نتيجة القبول:
□ كل المهام في < 5 دقائق → ناجح
□ 5-10 دقائق → مقبول
□ > 10 دقائق → يحتاج تحسين
```

## 4. استبيان الرضا (SUS — System Usability Scale)

```
لكل مشارك بعد التجربة:

1. أعتقد أنني سأستخدم هذا النظام بشكل متكرر
2. وجدت النظام معقداً أكثر من اللازم
3. وجدت النظام سهل الاستخدام
4. أعتقد أنني سأحتاج مساعدة فنية لاستخدام هذا النظام
5. وجدت أن وظائف النظام متكاملة بشكل جيد
6. هناك تناقض كبير في الوظائف
7. أتخيل أن معظم الناس سيتعلمون استخدام هذا النظام بسرعة
8. وجدت النظام صعب الاستخدام
9. شعرت بثقة أثناء استخدام النظام
10. احتجت تعلم الكثير قبل البدء

كل سؤال: 1 (أوافق بشدة) → 5 (لا أوافق بشدة)
حساب SUS: (مجموع الأسئلة الفردية - 5) + (25 - مجموع الأسئلة الزوجية) × 2.5
النتيجة من 100.
```

### معايير القبول

| SUS Score | التقييم | القرار |
|-----------|---------|--------|
| > 80 | ممتاز ✅ | إطلاق |
| 70-80 | جيد ✅ | إطلاق بعد تحسينات طفيفة |
| 50-69 | مقبول 🟡 | إطلاق مشروط + تحسينات |
| < 50 | ضعيف ❌ | تعليق الإطلاق + إعادة تصميم |

## 6. قائمة ما قبل UAT

```
□ 10-20 مشارك متنوع (أعمار، خلفيات، أجهزة)
□ رابط التجربة جاهز (Staging)
□ حسابات اختبار جاهزة (Client + Employee + Admin)
□ بيانات اختبار وهمية (10+ طلبات في حالات مختلفة)
□ كل الشاشات مترجمة للعربية
□ الأجهزة المطلوبة: جوالات Android + iOS + Laptop
□ اتصال إنترنت (4G + Wi-Fi)
□ منسق الجلسة جاهز
□ نماذج الموافقة للتصوير/التسجيل
□ استبيان SUS جاهز
□ مكافآت المشاركين (إن وجدت)
```
