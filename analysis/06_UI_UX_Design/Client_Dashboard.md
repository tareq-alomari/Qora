### مبدأ الواجهة الواحدة (Single Backend)

> **الموقع الرئيسي ولوحة التحكم يشتركان في نفس الـ Backend API.**
> الفرق الوحيد هو:
> - **الـ Role**: Client → يرى مسارات معينة. Employee → يرى مسارات إضافية. Admin → يرى كل شيء.
> - **الـ UI**: واجهة مختلفة (URL مختلف)، لكن خلفها نفس API.

```
الموقع الرئيسي:   https://qor3a.com           ← للعملاء
لوحة الموظفين:    https://qor3a.com/dashboard  ← للموظفين والمدير
```

## 2. Roles & Access Control

### 2.1 الـ Roles الثلاثة

| الـ Role | الوصف | الصلاحيات |
|----------|-------|-----------|
| **Client** | العميل المسجل | يرى طلباته فقط، يدفع، يفحص نتيجته |
| **Employee** | موظف المكتب | يرى كل الطلبات، يراجع، يدخل Confirmation# |
| **Admin** | مدير النظام | يرى كل شيء، يدير المستخدمين، يصدر التقارير |

### 2.2 توزيع المسارات حسب الـ Role

```
/login                          ← أي أحد (Client, Employee, Admin)
/register                       ← أي أحد
/lottery/*                      ← Client فقط (خطوات التسجيل)
/dashboard                      ← Employee + Admin (يدخلون فوراً)
/dashboard/admin/*              ← Admin فقط

/api/v1/orders                  ← Client (طلباته فقط) / Employee (الكل)
/api/v1/admin/*                 ← Employee + Admin
/api/v1/admin/users             ← Admin فقط
/api/v1/admin/reports           ← Admin فقط
```

### 2.3 Flow: كيف يدخل الموظف/المدير؟

```
1. يفتح الموقع → الصفحة الرئيسية
2. يضغط "دخول الموظفين" (زر صغير في التذييل أو /admin-login)
3. يدخل بريده الوظيفي + كلمة سر
4. النظام يكتشف role = employee/admin
5. يعيد التوجيه مباشرة إلى /dashboard
6. لا يرى أبداً واجهة التسجيل العادية
```

## 3. Employee Dashboard - التحليل الكامل

### 3.1 وصف عام

لوحة الموظفين هي **قلب العملية التشغيلية**. الموظف يقضي معظم وقته هنا:
- يراجع الطلبات
- يتحقق من الصور والبيانات
- يؤكد الدفعات
- يدخل أرقام التأكيد

### 3.2 هيكل الصفحات

```
/dashboard
│
├── /                     ← الصفحة الرئيسية للموظف (الإحصائيات السريعة)
│
├── /orders               ← 🔴 أهم صفحة - كل الطلبات
│   ├── /                  ← قائمة بكل الطلبات (مرشحة)
│   └── /:id              ← عرض طلب واحد + إجراءات
│
├── /payments             ← إدارة الدفعات
│   └── /:id              ← تأكيد الدفع
│
├── /submit               ← إدخال Confirmation# (بعد الاعتماد)
│   └── /:id              ← إدخال الرقم + لقطة الشاشة
│
├── /check-results        ← فحص النتائج (قائمة بانتظار الفحص)
│   └── /:id              ← فحص طلب واحد
│
└── /profile              ← الملف الشخصي للموظف
```

## 5. Technical Implementation

### 5.1 Authentication Flow للوحة التحكم

```
الموظف/المدير
     │
     ▼
GET /login → يكتب البريد + كلمة السر
     │
     ▼
POST /api/v1/auth/login
     │
     ├── user.role === 'client' → رفض (هذه لوحة الموظفين)
     │
     ├── user.role === 'employee' → token مع role:employee
     │   └── إعادة توجيه إلى /dashboard
     │
     └── user.role === 'admin' → token مع role:admin
         └── إعادة توجيه إلى /dashboard/admin
```

### 5.2 Real-time Updates (WebSocket)

الموظف يحتاج أن يرى الطلبات الجديدة فوراً (بدون تحديث الصفحة يدوياً).

```
استراتيجية: Server-Sent Events (SSE) أو WebSocket

حدث: طلب جديد
     └── → يظهر بطاقة حمراء في Dashboard:
          "🆕 طلب جديد من أحمد محمد - بانتظار المراجعة"

حدث: تأكيد دفع
     └── → تحديث العداد في الـ Sidebar:
          💳 4 (بدلاً من 5) بانتظار التأكيد
```

## 8. ملاحظات إضافية

### 🔴 مهم للواجهة
1. **الـ Sidebar يختلف حسب الـ Role**: Employee لا يرى "إدارة المستخدمين" أو "التقارير".
2. **الطلبات الجديدة**: تظهر مع تأثير (Animation) لفت انتباه الموظف.
3. **بحث سريع**: شريط بحث يعمل بـ Full-Text Search على اسم العميل ورقم الطلب.

### 🟡 مهم للتجربة
1. كل إجراء (قبول، رفض، تأكيد دفع) **لا يحتاج تحديث الصفحة** - AJAX + تحديث فوري للقائمة.
2. الـ Sidebar يظهر عداداً بعدد الطلبات المنتظرة في كل قسم.
3. لوحة المدير تحتوي على **Dashboard قابل للتخصيص** (يحذف/يضيف كروت).

### 🟢 مهم للتقنية
1. استخدم **React.lazy + Suspense** لكل صفحة في لوحة التحكم - الموظف لا يحتاج تحميل كل الصفحات دفعة واحدة.
2. الـ WebSocket يربط فقط لوحة التحكم - لا يربط الموقع الرئيسي (توفير موارد).
3. كل طلب API للوحة التحكم يحتاج تحقق من الـ Role في الـ Middleware (وليس فقط في الـ UI).
