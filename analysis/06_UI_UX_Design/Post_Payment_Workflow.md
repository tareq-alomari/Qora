# تحليل آلية ما بعد الدفع: من التأكيد إلى التسليم وفحص النتيجة

## Qor3a - Post-Payment Operations Workflow

---

## 1. نظرة عامة على التدفق الكامل

```
يدفع العميل ← موظف يؤكد ← يدخل الموقع الرسمي ← يحصل على Confirmation#
    → يسلم للعميل ← ينتظر مايو ← يفحص النتيجة ← يخبر العميل
```

### دورة حياة الطلب (حالات الـ State Machine)

```
payment_verification
    │
    ├── ✅ موظف يؤكد الدفع → approved
    │       │
    │       ├── ✅ موظف يدخل بيانات العميل في الموقع الرسمي ويحصل على Confirmation#
    │       │       │
    │       │       ├── submitted (حُفظ رقم التأكيد)
    │       │       │       │
    │       │       │       ├── اكتمل الإرسال للعميل → completed
    │       │       │       │       │
    │       │       │       │       ├── مايو: فحص النتيجة → won / lost
    │       │       │       │
    │       │       │       └── فشل الإدخال الرسمي → needs_correction
    │       │
    │       └── ❌ الموقع الرسمي يواجه مشكلة → موظف يحاول لاحقاً
    │
    ├── 📝 يحتاج تعديل → needs_correction (يعود للعميل)
    │
    └── ❌ رفض → cancelled
```

## 2. التدفق المفصل خطوة بخطوة

### المرحلة 1: تأكيد الدفع (Payment Verification)
```
عندما يرفع العميل صورة الإشعار → تظهر في لوحة الموظفين

الموظف يفتح الطلب ← يرى:
  ├── صورة الإشعار
  ├── المبلغ (1,000 YR)
  ├── طريقة الدفع (كريمي/جيب/ون كاش/موبايل موني)
  ├── اسم المحول (العميل)
  └── التاريخ

الإجراءات المتاحة:
  ├── ✅ "تأكيد الدفع" → المبلغ = السعر → approved
  ├── 📝 "المبلغ ناقص" → يكتب ملاحظة → needs_correction
  └── ❌ "إشعار مزور" → يرفض → cancelled

⚠️ لا يوجد API للتحقق من الحوالات - كل شيء يدوياً
```

### المرحلة 2: إدخال Confirmation# (التحضير للتسجيل الرسمي)

> **هذه أهم مرحلة في العملية كلها**. الموظف الآن يجب أن يدخل بيانات العميل في الموقع الرسمي لوزارة الخارجية الأمريكية (DV Lottery).

```
الموظف يراجع بيانات العميل ← يتأكد من صحتها 100%
  │
  ├── ✅ البيانات صحيحة ← يفتح الموقع الرسمي يدوياً
  │
  └── ❌ خطأ في البيانات ← needs_correction ← يعيد للعميل

عند فتح الموقع الرسمي، الموظف يقوم بـ:
  1. يدخل إلى https://dvprogram.state.gov/
  2. يسجل الدخول (إن لزم)
  3. يختار "Enter Confirmation"
  4. يدخل البيانات بالكامل يدوياً
     - الاسم الكامل
     - تاريخ الميلاد
     - الجنسية
     - رقم جواز السفر
     - الصورة (يرفع الصورة من جهازه)
     - العنوان
     - المؤهل
     - ...
  5. يضغط Submit
  6. يحصل على Confirmation Number 🎯
  7. يأخذ Screenshot أو نسخة PDF من صفحة التأكيد

⚠️ الموقع الرسمي لا يوفر API - الإدخال يدوي بالكامل
⚠️ الموقع الرسمي له CAPTCHA خاص به أثناء الإدخال
```

### تحديات الإدخال الرسمي (Manual Entry)

| التحدي | التأثير | الحل في قرعة |
|--------|---------|-------------|
| الموقع الرسمي بطيء | الموظف ينتظر تحميل الصفحات | لا حل - صبر |
| CAPTCHA في الموقع الرسمي | يحمي من البوتات ويبطئ الإدخال | الموظف يحلها يدوياً (هو أصلاً بشر) |
| خطأ في الإدخال اليدوي | رفض الطلب من اللوتري | مراجعة مضاعفة للبيانات قبل الإدخال |
| الموقع الرسمي ينهار وقت الذروة | لا يمكن الدخول | المحاولة في أوقات متفرقة |
| انقطاع الإنترنت في اليمن | فقدان العمل | حفظ البيانات محلياً وإعادة المحاولة |

### المرحلة 3: تسجيل Confirmation# في النظام
```
بعد أن يدخل الموظف البيانات في الموقع الرسمي ويحصل على Confirmation#:

الموظف يعود إلى لوحة قرعة ← يفتح الطلب
  │
  ├── يُدخل Confirmation Number في الحقل المخصص
  ├── يرفع Screenshot/PDF من صفحة التأكيد (اختياري)
  ├── يضغط "حفظ وإرسال إشعار للعميل"
  │
  ├── → حالة الطلب: submitted
  └── → إشعار فوري للعميل: "🎉 تم تسجيل طلبك! رقم التأكيد: 2026AB1234567"
```

### المرحلة 4: تسليم رقم التأكيد للعميل (Delivery)

#### قنوات الإرسال:
```
1. 📱 واتساب ← نص + صورة التأكيد (إن وجدت)
2. 📧 إيميل ← تفاصيل كاملة
3. 🔔 إشعار في حساب العميل على الموقع
4. 📄 قابل للتحميل (PDF) ← من حساب العميل
```

#### محتوى الإشعار:
```
🎉 تم تسجيلك في قرعة DV Lottery 2026!

رقم التأكيد: 2026AB1234567
الاسم: أحمد محمد عبدالله
تاريخ الميلاد: 15/05/1990

📌 احتفظ بهذا الرقم - ستحتاجه لفحص النتيجة في مايو 2026

🔍 يمكنك فحص النتيجة تلقائياً من حسابك في قرعة:
[رابط فحص النتيجة]

شكراً لثقتك بقرعة 🙏
```

### المرحلة 5: اكتمال الطلب (Completed)
```
بعد إرسال الإشعار للعميل ← حالة الطلب: completed

العميل يمكنه الآن:
  ├── عرض رقم التأكيد في أي وقت من حسابه
  ├── تحميل صورة التأكيد (PDF)
  ├── طلب فحص النتيجة (عند إعلان مايو)
  └── إنشاء طلب جديد (للأهل/الأصدقاء)
```

| ⏳ إجمالي الوقت من الدفع إلى التسليم | < يوم عمل واحد | في موسم الذروة قد يطول |

## 5. فحص النتيجة (Result Checking) - مايو 2026

### متى؟
كل سنة في **أول أسبوع من مايو** تُعلن نتائج DV Lottery.

### كيف تعمل في قرعة؟

```
العميل (أو الموظف) يضغط "🔍 فحص النتيجة"
    │
    ▼
1. النظام يقرأ Confirmation# من قاعدة البيانات
    │
    ▼
2. Headless Browser (Puppeteer) يفتح الموقع الرسمي
    │
    ▼
3. يدخل Confirmation# + البيانات الشخصية
    │
    ▼
4. CAPTCHA يظهر → 2Captcha أو يدوي
    │
    ▼
5. النتيجة:
    ├── 🎉 "Selected" → WINNER
    │   ├── إشعار فوري للعميل
    │   ├── فريق قرعة يتواصل مع العميل
    │   └── حالة الطلب: winner
    │
    └── 😢 "Not Selected" → LOSER
        ├── إشعار للعميل: "لم يتم اختيارك"
        └── حالة الطلب: completed
```

### التحديات في فحص النتيجة

| التحدي | التأثير | الحل |
|--------|---------|------|
| **آلاف الطلبات** | 500+ عميل يريدون الفحص فوراً | Queue + Workers (5-10 متوازيين) |
| **CAPTCHA معقد** | 2Captcha قد يفشل | طابور يدوي احتياطي + Proxy Rotation |
| **الموقع الرسمي ينهار** | لا يمكن الوصول | جدولة ذكية (أوقات متفرقة) |
| **حظر IP الخادم** | وقف الفحص بالكامل | Proxy Pool + تأخير عشوائي بين الطلبات |

### معالجة آلاف الطلبات (Queue Strategy)

```
إعلان النتائج (1 مايو)
    │
    ▼
┌────────────────────────────────────┐
│  500+ عميل يضغط "فحص النتيجة"      │
└──────────────┬─────────────────────┘
               │
               ▼
┌────────────────────────────────────┐
│  Queue (Redis/Bull)               │
│  ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐       │
│  │ط1│ │ط2│ │ط3│ │ط4│ │ط5│ ...   │
│  └──┘ └──┘ └──┘ └──┘ └──┘       │
└──────────────┬─────────────────────┘
               │
               ▼
┌────────────────────────────────────┐
│  Worker Pool (10 Workers)         │
│  ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐      │
│  │W1│ │W2│ │W3│ │W4│ │W5│ ...  │
│  └──┘ └──┘ └──┘ └──┘ └──┘      │
│  كل Worker:                      │
│  1. يفتح Headless Browser       │
```

## 7. حالات الطلب النهائية وماذا تعني للعميل

| الحالة | معناها | يظهر للعميل |
|--------|--------|-------------|
| `approved` | الدفع مؤكد، بانتظار الإدخال الرسمي | "قيد المعالجة" |
| `submitted` | تم إدخال البيانات رسمياً + Confirmation# محفوظ | "🎉 تم التسجيل! رقم التأكيد: XXXXX" |
| `completed` | اكتمل التسجيل وتم إرسال الإشعار للعميل | "✅ مكتمل - اضغط لفحص النتيجة" |
| `winner` | 🎉 العميل فاز باللوتري | "🎉 مبروك! تم اختيارك!" |
| `cancelled` | أُلغي الطلب | "تم إلغاء الطلب" |
| `needs_correction` | يحتاج تعديل من العميل | "📝 نحتاج منك تعديل البيانات" |

## 8. متطلبات تقنية إضافية

### للإدخال الرسمي:
- الموظف يحتاج **اتصال إنترنت مستقر** للدخول على travel.state.gov
- **VPN** قد يكون ضرورياً (الموقع الرسمي قد يكون محجوباً في اليمن)
- الموظف يحتاج **متصلب حديث** (Chrome/Firefox) للتعامل مع الموقع الرسمي

### للفحص الآلي:
- **Headless Browser Cluster** (5-10 Workers)
- **Proxy Pool** (10-20 IPs) لتجنب الحظر
- **2Captcha API Key** لحل CAPTCHA
- **Redis Queue** لإدارة طابور الفحص
- **Rate Limiting** (لا تفحص أكثر من طلبين في الدقيقة من نفس IP)

### للإشعارات:
- **WhatsApp Business API** (أو WATI/UltraMsg للوسطاء)
- **SMTP** (للإيميلات)
- **WebSocket** (للإشعارات الفورية في لوحة التحكم)
