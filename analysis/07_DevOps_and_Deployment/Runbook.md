# استمرارية المؤسس - Founder Business Continuity

## Qor3a — ماذا لو اختفى المؤسس؟

---

## 1. المشكلة

```
أنت (المؤسس) تملك:
├── 🖥️ Access إلى السيرفر (SSH)
├── ☁️ Access إلى Cloudflare (DNS)
├── 🌐 الـ Domain (Namecheap/Cloudflare)
├── 💳 حسابات الدفع (كريمي، جيب، ون كاش)
├── 🗄️ قاعدة البيانات (PostgreSQL password)
├── 🔑 API Keys (2Captcha, WhatsApp, Meta)
├── 📱 رقم واتساب الدعم (مرتبط بحسابك الشخصي)
├── 💰 الفلوس (حسابات Qor3a البنكية)
├── 🐙 GitHub repo (الـ code)
└── 📧 إيميل الإدارة (info@qor3a.com)

لو اختفيت أنت لأي سبب:
├── 🚑 حادث
├── 🏥 مرض
├── 🌐 انقطاع الإنترنت (في اليمن)
├── 🔌 انقطاع الكهرباء (طويل)
└── أي سبب يمنعك من الوصول للإنترنت لأكثر من أسبوع

ماذا يحدث؟
├── ❌ لا أحد يستطيع دخول السيرفر
├── ❌ لا أحد يستطيع تأكيد الدفع (الفلوس في الحساب)
├── ❌ لا أحد يستطيع تجديد Domain (الموقع يطير)
├── ❌ لا أحد يستطيع دفع فاتورة السيرفر
├── ❌ لا أحد يستطيع الوصول للكود
└── ❌ Qor3a تتوقف تماماً
```

---

## 2. الحل: الـ Runbook (دفتر التشغيل الطارئ)

### 2.1 يجب أن يكون هذا الملف موجوداً في 3 أماكن

```
الملف: /opt/qor3a/RUNBOOK.md (على السيرفر)
النسخ:
├── 📄 1. على السيرفر (مشفر)
├── 📄 2. عند شخص موثوق (مشفر)
└── 📄 3. في مكان آمن خارجي (مشفر — مثلاً 1Password)

لا تضع كل شيء في ملف واحد غير مشفر!
استخدم مدير كلمات سر (Password Manager).
```

### 2.2 ماذا يحتوي الـ Runbook؟

```
📋 RUNBOOK — Qor3a Emergency Operations
═══════════════════════════════════════

🔐 الصفحة 1: الوصول في حالات الطوارئ

1. الوصول إلى السيرفر:
   ├── SSH: root@SERVER_IP
   ├── Port: 22
   ├── Key: /path/to/private-key (عند الموثوق)
   └── Password Manager: 1Password / Bitwarden

2. الوصول إلى Domain:
   ├── المسجل: Namecheap
   ├── الإيميل: admin@qor3a.com
   └── Password Manager: 1Password / Bitwarden

3. الوصول إلى Cloudflare:
   ├── Account: admin@qor3a.com
   └── Password Manager: 1Password / Bitwarden

4. الوصول إلى GitHub:
   ├── Organization: Qor3a
   ├── Admin User: [اسم المستخدم]
   └── Password Manager: 1Password / Bitwarden

5. الوصول إلى حسابات الدفع:
   ├── 🏦 كريمي: رقم الحساب، PIN (في Manager)
   ├── 🏦 جيب: رقم الحساب، PIN
   ├── 🏦 ون كاش: رقم الحساب، PIN
   └── 🏦 موبايل موني: رقم الحساب، PIN

6. API Keys:
   ├── 2Captcha: KEY (في Manager)
   ├── WhatsApp Cloud API: TOKEN
   ├── SendGrid: API KEY
   ├── Meta Developer: TOKEN
   └── SMTP: PASSWORD

7. قاعدة البيانات:
   ├── DB_HOST: localhost
   ├── DB_NAME: qor3a
   ├── DB_USER: postgres
   └── DB_PASSWORD: (في Manager)
```

### 2.3 من له حق الوصول للـ Runbook؟

```
المستوى 1: المؤسس (الوصول الكامل)
├── كل شيء

المستوى 2: الشريك التقني (وصول جزئي)
├── ✅ السيرفر (SSH)
├── ✅ GitHub
├── ✅ API Keys
├── ✅ قاعدة البيانات
├── ❌ Domain (نقل الملكية)
├── ❌ حسابات الدفع (سحب فلوس)
└── ❌ Cloudflare (تغيير DNS)

المستوى 3: موظف موثوق (وصول محدود جداً)
├── ❌ لا شيء مباشر
├── ✅ عنده رقم الطوارئ
├── ✅ يعرف من يتصل
└── ✅ يشتغل لحين عودة المؤسس

ملاحظة:
├── المستوى 2 يصبح مستوى 1 فقط في حالة طارئة (بإجماع 2+)
└── مثلاً: إذا المؤسس غائب > 7 أيام بدون تواصل
```

## 3. التوزيع الجغرافي للمفاتيح

```
المؤسس (اليمن):
├── 🖥️ حاسوب شخصي (يحتوي مفاتيح SSH)
├── 📱 جوال (2FA, واتساب بزنس)
└── 📄 ورقة مطبوعة (في خزينة — مفاتيح الطوارئ)

الشريك التقني (خارج اليمن — يفضل أمريكا/أوروبا):
├── 🖥️ سيرفر خاص (نسخة احتياطية من DB)
├── 📱 جوال (2FA احتياطي)
└── 🗄️ Password Manager (مشارك مع المؤسس)

الموظف (اليمن):
├── 📱 جوال (رقم واتساب الدعم)
└── 📞 أرقام الطوارئ

موثوق خارجي (صديق/محامي — خارج اليمن):
├── 📄 ظرف مختوم: "يُفتح فقط إذا المؤسس غائب 7+ أيام"
├── 🗄️ يحتوي: Domain credentials + instructions
└── 📞 أرقام كل الأطراف
```

## 4. إجراءات الطوارئ (من يتصل بمين وبأي ترتيب)

```
عند انقطاع المؤسس عن العمل لأكثر من 24 ساعة:

الموظف:
├── 📞 يتصل بالمؤسس (واتساب + اتصال)
├── ⏳ ينتظر 24 ساعة
├── 📞 يتصل بالشريك التقني
└── 📞 يبلغ الأطراف المعنية

الشريك التقني (عند الإبلاغ):
├── 🔐 يدخل Password Manager
├── 🖥️ يتفقد السيرفر (هل كل شيء شغال؟)
├── 💳 يتفقد حسابات الدفع (هل فيه معاملات معلقة؟)
├── 📧 يرد على استفسارات العملاء
└── ⏳ ينتظر 7 أيام قبل أي قرار كبير

بعد 7 أيام (إذا لا زال المؤسس غائباً):
├── 🔄 نقل Domain إلى حساب الشريك
├── 💰 تفعيل الدفع لرواتب الفريق
├── 📋 إشعار للعملاء: "خدمة متوقفة مؤقتاً"
└── 🤝 البحث عن بديل للمؤسس
```

## 5. ماذا يحتاج الفريق ليكمل بدونك؟

### 5.1 الحد الأدنى من المعلومات (Minimum Viable Runbook)

```
إذا أردت فقط 5 أشياء يكتبها الفريق (لأنه لا يمكنهم إدارة كل شيء بدونك):

1️⃣ 🐙 GitHub: مصدر الكود (يتغير)
2️⃣ 🖥️ السيرفر: SSH key موجودة في الـ Password Manager
3️⃣ 💳 حسابات الدفع: PIN موجودة + الموظف يقدر يشوف الحوالات
4️⃣ 🏛️ Domain: في Namecheap مع auto-renew
5️⃣ 📱 واتساب بزنس: الرقم ليس رقمك الشخصي (رقم منفصل)
```

### 5.2 الأسئلة الحرجة

```
هل الفريق يستطيع:
├── ❓ دخول السيرفر؟ (SSH key)
├── ❓ رؤية الطلبات الجديدة؟ (تلقائي)
├── ❓ تأكيد الدفع؟ (يحتاج حساب كريمي/جيب)
├── ❓ صرف رواتب؟ (يحتاج حساب بنكي)
├── ❓ تجديد الـ Domain؟ (يحتاج Namecheap)
└── ❓ التواصل مع العملاء؟ (يحتاج واتساب)

كل إجابة بـ "لا" = خطر.
```

## 6. الحلول العملية (باش نبدأ ننفذ)

### 6.1 اليوم

```
□ فتح Password Manager (1Password أو Bitwarden — $3/شهر)
□ إضافة كل كلمات السر (Domain, Server, API Keys, Email)
□ دعوة الشريك التقني إلى Password Manager (قراءة فقط)
□ إعداد 2FA (Authenticator app — مشاركة Secret مع الشريك)
□ تجديد Domain تلقائياً (auto-renew + بطاقة ائتمان)
□ تحويل رقم واتساب بزنس إلى رقم منفصل (ليس رقمك الشخصي)
```

### 6.2 هذا الأسبوع

```
□ كتابة RUNBOOK.md ورفعه إلى السيرفر
□ طباعة نسخة ورقية في ظرف مختوم
□ إعطاء الظرف لشخص موثوق خارج اليمن
□ تحديد الشريك التقني (المستوى 2)
□ شرح الخطة للفريق: "إذا ما رديت 24 ساعة، اتصل بـ X"
□ اختبار الطوارئ: فصل المؤسس لمدة يوم (متعمد) هل الفريق يكمل؟
```

### 6.3 هذا الشهر

```
□ تفعيل GitHub Organization (وليس حساب شخصي)
□ إضافة Admin ثاني على المنصة (لوحة التحكم)
□ إعداد بطاقة ائتمان مشتركة لفواتير السيرفر
□ إعداد تحويل تلقائي للرواتب
□ توثيق كل شيء في الـ RUNBOOK
```
