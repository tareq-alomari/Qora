## 2. خطة الطوارئ لكل سيناريو

### 2.1 🔴 انقطاع السيرفر (Server Down)

```
العلامات:
├── الموقع لا يفتح
├── خطأ 502/503
└── الـ Ping يفشل

الإجراء الفوري:
1️⃣ تحقق من الـ Dashboard (UptimeRobot)
2️⃣ ادخل على السيرفر عبر SSH
3️⃣ شوف الـ Logs: docker compose logs
4️⃣ سبب المشكلة:
    ├── زيادة تحميل → إعادة تشغيل الخدمة + زيادة الموارد
    ├── نفاد مساحة → تنظيف Logs + زيادة التخزين
    └── تحديث فاشل → الرجوع للإصدار السابق

الإجراء إذا السيرفر نفسه مات:
1️⃣ شغّل السيرفر الاحتياطي
2️⃣ غير DNS إلى IP السيرفر الاحتياطي
3️⃣ استرجع آخر Backup للـ DB
4️⃣ الخدمة تعود خلال 30-60 دقيقة

الوقت اللازم: 30-60 دقيقة
التنبيه: إيميل + تيليجرام فوري
```

### 2.2 🔴 تلف قاعدة البيانات (Data Corruption)

```
العلامات:
├── بيانات ناقصة
├── علاقات (Foreign Keys) مضروبة
└── أخطاء في استعلامات

الإجراء:
1️⃣ إيقاف الخدمة (qor3a.com يعرض صفحة "تحت الصيانة")
2️⃣ تحديد Backup: آخر نسخة صحيحة
3️⃣ استرجاع:
    docker exec qor3a-postgres psql -U postgres qor3a < backup_FILE.sql
4️⃣ التحقق من سلامة البيانات
5️⃣ تشغيل الخدمة

💡 نصيحة: اختبر استرجاع الـ Backup كل شهر للتأكد من أنه سليم!

الوقت اللازم: 1-2 ساعة
التنبيه: إيميل + تيليجرام فوري
```

### 2.3 🟡 انقطاع الإنترنت في اليمن

```
المشكلة:
├── العملاء اليمنيون لا يستطيعون الوصول للموقع
└── لكن السيرفر خارج اليمن (يعمل عادي)

الحل:
├── ✅ لا يوجد ضرر على البيانات
├── ✅ الطلبات قيد المعالجة مستمرة
├── ❌ العملاء الجدد لا يستطيعون التسجيل
└── ⏸️ بعد عودة الإنترنت: العملاء يكملون من حيث توقفوا

تحديث الحالة:
├── إرسال إشعار للعملاء الحاليين: "الخدمة متوقفة مؤقتاً في اليمن"
└── رسالة على الموقع: "نأسف، الخدمة متوقفة حالياً في اليمن"

الوقت اللازم: غير معروف (حسب انقطاع الإنترنت)
```

### 2.4 🔴 اختراق أمني (Security Breach)

```
العلامات:
├── تسجيل دخول مشبوه
├── طلبات غير معروفة
├── تعديل بيانات بدون تصريح
└── إشعار من مزود الاستضافة

الإجراء الفوري:
1️⃣ إيقاف الخدمة فوراً
2️⃣ قطع اتصال السيرفر بالإنترنت (أو فصل الـ Database)
3️⃣ تغيير كل كلمات السر:
    ├── DB Password
    ├── Admin Passwords
    ├── JWT Secret
    ├── API Keys
    └── SSH Keys
4️⃣ تحليل الـ Logs: من؟ متى؟ ماذا حدث؟
5️⃣ تقييم الضرر: هل تسربت بيانات العملاء؟
6️⃣ إبلاغ العملاء المتضررين (إيميل/واتساب)
7️⃣ استرجاع من Backup قبل الاختراق (إن لزم)
8️⃣ تشغيل الخدمة بعد التأكد من إغلاق الثغرة

الوقت اللازم: 4-24 ساعة
التنبيه: إيميل + تيليجرام + اتصال هاتفي فوري
```

### 2.5 🟡 فشل AI Service

```
العلامات:
├── /api/ai/health → غير 200
├── طلبات الصور تتراكم
└── Error Logs في AI Service

الإجراء:
1️⃣ تفعيل الوضع اليدوي: كل الصور ترسل للموظف لفحصها يدوياً
2️⃣ شوف Logs الـ AI: خطأ في المكتبة؟ نموذج فاشل؟
3️⃣ إعادة تشغيل الخدمة:
    docker compose restart ai-service
4️⃣ إذا استمرت المشكلة:
    ├── الرجوع لآخر إصدار مستقر من AI Service
    └── أو إصلاح المشكلة في الكود

الوقت اللازم: 15-60 دقيقة
```

## 3. قائمة التحقق الأسبوعية (Weekly Health Check)

```
□ تحقق من Storage: df -h (هل التخزين < 80%؟)
□ تحقق من الـ Backup: هل تم أمس؟
□ تحقق من الـ Logs: أي أخطاء متكررة؟
□ تحقق من SSL: هل سينتهي قريباً؟
□ تحقق من API Health: كل الخدمات شغالة؟
□ تحقق من الـ Queue: أي طلبات عالقة؟

تستغرق: 5-10 دقائق
المسؤول: المطور أو مشرف النظام
```

## 4. قائمة التحقق الموسمية (قبل موسم التسجيل)

```
قبل موسم التسجيل بـ 30 يوماً:
□ هل الاختبارات كلها تمر؟ (npm test)
□ هل الـ Backup Strategy شغالة؟
□ هل السيرفر الاحتياطي جاهز؟
□ هل الـ Rate Limiting مضبوط؟
□ هل الـ Proxy/VPN للـ Headless Browser يعمل؟
□ هل حساب 2Captcha مشحون؟
□ هل كل API Keys صالحة؟
□ هل SSL شهادة سارية لـ 6+ أشهر؟
□ Page Speed للموقع > 70 (على الهاتف)
□ اختبار تحميل: الموقع يتحمل 500 مستخدم معاً؟
```

## 5. الخطة بعد الكارثة (Post-Mortem)

```
بعد أي مشكلة كبيرة، يجب عمل:
1️⃣ تقرير: ماذا حدث بالضبط؟
2️⃣ سبب جذري: لماذا حدث؟
3️⃣ حل مؤقت: كيف تم حل المشكلة؟
4️⃣ حل دائم: كيف نمنع تكرارها؟
5️⃣ دروس مستفادة: ماذا تعلمنا؟

مثال نموذج Post-Mortem:
━━━━━━━━━━━━━━━━━━━━━━
📋 Post-Mortem: انقطاع 3 ساعات
📅 التاريخ: 12 يوليو 2026
⏰ المدة: 3 ساعات

🩺 السبب: نفاد مساحة التخزين
├── Logs ملأت المساحة (لم نضبط logrotate)

🚑 الحل المؤقت:
├── حذف الـ Logs القديمة
├── تشغيل الخدمة

🏗️ الحل الدائم:
├── تفعيل logrotate مع ضغط يومي
├── تنبيه تلقائي عند 70% مساحة

📚 الدروس:
├── نضيف مراقبة التخزين
├── نختبر الاسترجاع من Backup كل شهر
━━━━━━━━━━━━━━━━━━━━━━
```

## 6. الـ SLA (Service Level Agreement)

### 6.1 وعدنا للعملاء

| المقياس | الهدف |
|---------|-------|
| وقت التشغيل (Uptime) | > 99.5% شهرياً |
| زمن استجابة API | < 500ms (في اليمن) |
| وقت معالجة الصورة (AI) | < 5 ثوانٍ |
| وقت تأكيد الدفع | < 4 ساعات |
| وقت الرد على الاستفسار | < 30 دقيقة (واتساب) |

### 6.2 ماذا نقدم إذا انقطعنا؟

```
أقل من 1 ساعة → لا شيء (normal operation)
1-4 ساعات → خصم 20% للطلبات المتأثرة
أكثر من 4 ساعات → خصم 50% + توصيل مجاني للخدمة
أكثر من 24 ساعة → استرداد كامل للطلبات المتأثرة
```
