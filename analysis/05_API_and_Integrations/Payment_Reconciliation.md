## 1. التحدي

```
العميل يحوّل 1,000 YR على حساب كريمي/جيب/ون كاش/موبايل موني.
ويرفع صورة الإشعار.

المشكلة:
├── ⚠️ الإيصال قد يكون مزوراً (فوتوشوب)
├── ⚠️ المبلغ قد يكون ناقصاً (950 بدلاً من 1,000)
├── ⚠️ إيصال مكرر (نفس الصورة لطلبين)
├── ⚠️ اسم المرسل مختلف (يحوّل من رقم غير مسجل)
└── ⚠️ العميل لم يرفع الإيصال أبداً (دفع ونسي)
```

---

## 2. تدفق التسوية (Reconciliation Flow)

```
العميل يرفع صورة الإشعار
        │
        ▼
  ⏳ Pending Verification
        │
        ▼
  الموظف يفتح صورة الإشعار
        │
        ├── 🔍 تحقق يدوي:
        │   1️⃣ هل المبلغ = 1,000 YR؟
        │   2️⃣ هل الإيصال حقيقي (غير معدّل)؟
        │   3️⃣ هل تاريخ التحويل اليوم؟
        │   4️⃣ هل الحساب المستلم = أحد حساباتنا؟
        │
        ├── ✅ صحيح → تأكيد الدفع
        │   ← الطلب ينتقل إلى payment_verification
        │
        ├── ❌ مشكلة → Needs Correction
        │   ← نرسل للعميل: "في مشكلة في الإيصال، يرجى إعادة الرفع"
        │   ← مع ذكر السبب (مبلغ ناقص، إيصال غير واضح...)
        │
        └── ⏳ غير متأكد → التحقق من حساب البنك
            ← نفتح تطبيق كريمي ونتأكد من المعاملة
            ← إذا أكيدة ← تأكيد
            ← إذا لا ← رفض
```

## 3. نظام مطابقة الإيصالات (Receipt Matching System)

### 3.1 الحقول التي نحتاجها من الإيصال

```
من صورة الإيصال، الموظف يدخل:
├── 💰 المبلغ (1,000 YR)
├── 📱 من رقم (رقم المرسل)
├── 🏦 إلى حساب (كريمي/جيب/ون كاش/موبايل موني — رقم الحساب)
├── 📅 التاريخ والوقت
└── 🔢 رقم العملية (إن وجد)

هذه المعلومات تقارن مع:
├── 📋 الطلب (للتأكد من أن العميل دفع المبلغ الصحيح)
└── 📊 سجل الحساب البنكي (للتأكيد)
```

### 3.2 نظام مكافحة الاحتيال (Anti-Fraud)

```
🚨 علامات الخطر (Red Flags):
├── إيصال بنفس رقم العملية لطلبين → رفض الثاني
├── إيصال بمبلغ مختلف (900 أو 1,100) → مراجعة
├── العميل يرفع نفس الصورة 3 مرات → spam
├── إيصال من الأمس (ليس اليوم) → مقبول (قد يتأخر الرفع)
├── رقم المرسل غير موجود في النظام → عادي (قد يحوّل من رقم ثاني)
└── إيصال بدقة منخفضة جداً → قد يكون مزوراً

🟢 علامات الثقة:
├── نفس رقم واتساب العميل في الإيصال
├── مبلغ مضبوط (1,000)
├── تاريخ اليوم
└── إيصال واضح (غير مشوش)
```

### 3.3 OCR (للمرحلة الثانية)

```
المستقبل (بعد الـ MVP):
├── OCR يقرأ من صورة الإيصال تلقائياً:
│   ├── المبلغ
│   ├── تاريخ التحويل
│   ├── رقم العملية
│   └── اسم البنك
├── يقارن مع الطلب
└── إذا تطابق → تأكيد تلقائي (بدون موظف)

التحدي:
├── 🔴 صعوبة OCR في اليمن (إيصالات كريمي/جيب مختلفة)
├── 🟡 دقة منخفضة (قد نضطر لمراجعة يدوية)
└── 🟢 تحسن مع الوقت (Machine Learning)
```

## 4. حسابات الدفع (Payment Accounts)

### 4.1 الهيكل

```
🏦 حسابات قرعة:

كريمي:
├── 📱 رقم الحساب: 0123456789
├── 👤 الاسم: Qor3a Yemen
└── 📊 الحد اليومي: 500,000 YR

جيب:
├── 📱 رقم الحساب: 9876543210
├── 👤 الاسم: Qor3a Yemen
└── 📊 الحد اليومي: 300,000 YR

ون كاش:
├── 📱 رقم الحساب: 5555555555
├── 👤 الاسم: Qor3a
└── 📊 الحد اليومي: 200,000 YR

موبايل موني:
├── 📱 رقم الحساب: 7777777777
├── 👤 الاسم: Qor3a
└── 📊 الحد اليومي: 200,000 YR
```

### 4.2 إدارة الحسابات

```
كل رقم حساب له:
├── 🆔 ID في قاعدة البيانات
├── 📊 رصيد الحساب المتوقع (من الطلبات المؤكدة)
├── 🔄 يسوي تسوية أسبوعية (نقارن الرصيد المتوقع بالفعلي)
└── 📋 سجل معاملات

الإجراء الأسبوعي:
├── 📱 نفتح تطبيق كريمي
├── 📊 نقارن المعاملات مع الطلبات المؤكدة
├── ❓ أي معاملة بدون طلب؟ (قد يكون عميل ما سجل)
├── ❓ أي طلب بدون معاملة؟ (قد يكون إيصال مزور)
└── 📋 تسوية + تقرير
```

## 5. حالات الدفع الخاصة

### 5.1 عميل دفع مرتين

```
السيناريو:
├── العميل دفع 1,000 YR
├── رفع الإيصال
├── الموظف لم يؤكد بعد (أو لم يرَ)
├── العميل قلق، دفع مرة أخرى!
└── الآن عنده إيصالين

الحل:
├── الموظف يؤكد أول إيصال
├── الإيصال الثاني يرفض (مكرر)
├── العميل: استرداد للإيصال الثاني
└── أو: رصيد للخدمة القادمة
```

### 5.2 عميل دفع مبلغ ناقص

```
السيناريو:
├── المبلغ: 950 YR (بدلاً من 1,000)
├── الموظف يلاحظ ← needs_correction
├── رسالة للعميل: "المبلغ ناقص 50 ريال، يرجى استكمال المبلغ"
└── العميل يكمل ← تأكيد
```

### 5.3 إيصال مزور

```
السيناريو:
├── عميل يعدّل صورة الإيصال (يغير المبلغ)
├── الموظف يفتح ← يلاحظ التعديل
├── رفض الدفع
├── الطلب ← cancelled
├── ملاحظة: "إيصال معدّل"
└── العميل محظور (fraud)
```
